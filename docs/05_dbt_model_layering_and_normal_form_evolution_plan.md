# dbt Model Layering And Normal Form Evolution Plan

## Goal

Evolve the local PostgreSQL/dbt model into a sustainable, layered, contract-driven analytical model after approved adapters have already loaded the `data_500k` dataset into PostgreSQL review/Silver tables.

This plan is not an adapter-loading plan, not a Databricks rollout plan, and not a business-question-first plan. Its purpose is to make the dbt model maintainable from global standards to entity-specific normalized outputs, with clear lineage, robust QA, and a path from Silver contracts toward stable Gold-ready facts, dimensions, and marts.

## Operating Focus

Run this plan as a dbt model architecture and relational normalization effort. PostgreSQL is the local execution engine. dbt Core is the transformation, lineage, testing, and documentation layer. Metadata contracts remain the source of modeling intent.

The starting point is:

```text
approved provider adapters -> local PostgreSQL review/Silver tables -> dbt source() -> dbt layered model
```

Do not reload source files, rewrite adapters, hand-patch PostgreSQL as durable model history, or treat dbt as an isolated SQL scratchpad.

## Required Skills

Use `dbt-layering-orchestrator` as the primary dbt architecture skill. Use `relational-normal-form-reviewer` for entity-by-entity 1NF, 2NF, 3NF, and 4NF review. Use `local-postgres-dbt-modeling-orchestrator` for local PostgreSQL/dbt execution discipline and model evolution evidence. Use `spec-test-generator` for contract and SQL QA coverage. Use `qa-evidence-reviewer` and `privacy-governance-reviewer` before claiming a category or entity is closed. Use `hitl-escalation-controller` only when ownership, meaning, clinical interpretation, financial interpretation, privacy risk, provider metadata, or business impact remains unresolved after deterministic probing.

## Filesystem Contract

Follow `metadata/model_specs/dbt/dbt_model_layering_contract.yaml` and keep dbt models organized from global to specific:

```text
dbt/models/
  sources.yml
  data_quality/
    standardization/
    completeness/
    consistency/
    validity/
    uniqueness/
    accuracy/
    timeliness/
  derived/
    <entity>/
      silver/
        landing/
        staging/
        standardized/
        normalized/
        conformed/
        dq/
      gold/
        facts/
        dimensions/
        marts/
        exchange/
        exports/
      schema.yml
```

Create only the subfolders that are needed. `silver/normalized` is not the only allowed Silver layer; it is the entity-decomposition layer. `silver/standardized` should hold deterministic format/code/time cleanup when the same entity needs a stable pre-normalization standard. `silver/conformed` should hold cross-entity or cross-provider alignment such as reusable provider, domain, unit, or code-system relationships. `silver/dq` should hold survivor candidates, QA helper projections, and issue-preserving reconciliation layers.

## Modeling Rules

Every dbt model must declare or inherit its grain. Facts must have event, period, row, or transaction grain. Dimensions must have stable descriptive grain. Marts must be derived from facts and dimensions and must not become hidden normalization layers.

Use `source()` only at the source boundary and `ref()` everywhere else. Do not repeat identical cleanup in many final models. Promote repeated deterministic cleanup to macros or upstream standardization models.

Provider metadata should move toward a conformed provider dimension when reused across entities. This includes provider slug, display name, timezone, source system family, and approved timezone policy. Patients may belong to one or more providers; do not collapse provider-scoped identity into a false global identity unless a matching contract is approved.

Do not infer missing values when the contract says null is the stable representation. When a value cannot be safely resolved but must remain queryable, expose both the normalized field and an explicit boolean or status flag such as `is_unresolved`, `has_conflict`, `is_missing_required_field`, or `requires_human_review`.

## Normal Form Review

Review each entity family independently before broad refactors. The expected review order is providers/domains, patients, coverage, encounters, conditions, medications, observations, and cost records, unless QA evidence shows a higher-risk dependency.

For each entity, evaluate:

```text
1NF: atomic fields, no repeated groups, no nested payload leaking into scalar columns.
2NF: all non-key fields depend on the whole declared grain.
3NF: descriptive fields do not depend on other descriptive fields that should be dimensions.
4NF: independent multivalued facts are separated into their own relations or bridges.
```

If the model needs a new lookup dimension, bridge, survivor candidate, or conformed table, first create a normalization probe under `metadata/model_specs/evolution/V0_5/` unless the change is already fully described by an approved contract. The probe must include candidate grain, keys, source fields, row accounting, lineage strategy, QA tests, rollback target, and HITL status.

## Data Quality Application

`dbt/models/data_quality/` is a governed diagnostic and standardization layer. It is not enough for this layer to report a problem. A drift category is only considered closed when the derived model path that users query consumes the relevant standardization through macros, refs, conformed dimensions, survivor candidates, null handling, or unresolved flags.

Use the contracts under `metadata/model_specs/data_quality/contracts/` to decide what can be resolved deterministically. Do not request HITL for blank-to-null cleanup, trim/lower/upper normalization, provider-scoped key generation, approved timezone conversion, approved decimal normalization, or explicit unresolved flags.

## Materialization Strategy

Use `ephemeral` for small reusable helper SQL that should not persist as a relation. Use `view` for lightweight auditable layers and most Silver transformations. Use `table` for expensive, repeatedly queried facts, dimensions, survivor outputs, marts, or outputs needed for reliable full-data QA. Use incremental materialization only after a stable unique key, late-arriving-data rule, and rollback strategy are approved.

Materialized views are not the default. Do not create physical relations simply because a CTE is convenient. Do not keep development-only exploratory relations in the final model tree.

## Lineage And Documentation

dbt lineage must show how each final fact, dimension, and mart depends on sources, standardization, normalization, conformance, and data quality decisions. Generate dbt artifacts after material changes and capture them only through approved local validation.

Schema YAML must document model purpose, grain, key fields, quality flags, and expected tests. If a model is intentionally temporary, diagnostic, or non-production, name it and place it so that purpose is visible.

## QA Gates

Run validation with `uv`. Minimum gates are:

```text
uv run --no-sync dbt parse --project-dir dbt --profiles-dir dbt
uv run --no-sync dbt run --project-dir dbt --profiles-dir dbt --select <changed selectors>
uv run --no-sync dbt test --project-dir dbt --profiles-dir dbt --select <changed selectors>
uv run --no-sync pytest tests/specs
uv run --no-sync pytest tests/sql
uv run --no-sync ruff check tests/specs tests/sql
```

When a change touches full-data quality behavior, run the relevant exhaustive SQL pytest against local PostgreSQL, not only fixture tests. Required QA dimensions include row accounting, key uniqueness, required completeness, valid domains, provider isolation, standardization outputs, UTC/local temporal behavior, survivor logic, referential integrity, lineage preservation, and no silent row loss.

## HITL Rules

HITL is required for unresolved semantics, provider ownership, clinical meaning, financial meaning, privacy exposure, default timezone attribution without contract, survivorship preferences that change business interpretation, or proposed field deprecation. HITL is not required for deterministic transformations already governed by a contract.

Every HITL question must identify the entity, field, provider scope, source examples, proposed dbt layer, affected downstream models, risk, and recommended default.

## Trace Logging

Append concise entries to `logs/local_runtime/local_runtime_certification.md` for each material dbt architecture change. Include `plan_id=05`, provider scope, entity, layer, contract touched, validation command, and remaining risk. Do not rewrite existing trace history.

## Expected Outputs

Expected outputs include updated skills, an updated dbt layering contract when needed, entity-level normalization probes, dbt model moves or additions under the approved filesystem, schema YAML updates, dbt tests, SQL pytest coverage, data quality remediation reports, local dbt artifacts, and trace log entries.

This plan does not require new business-question SQL. Gold candidates may be introduced only when they follow from normalized/conformed entity models and have robust QA.

## Definition Of Done

Plan 05 is done when the dbt model tree has a sustainable Silver/Gold filesystem, each entity has a documented normal-form review, data quality contracts are applied to the queryable derived outputs, provider/domain lookup opportunities are either implemented or explicitly deferred, dbt lineage is coherent, full local PostgreSQL QA passes for changed entities and categories, and remaining HITL decisions are explicit.
