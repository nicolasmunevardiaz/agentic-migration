# Relational Normal Form Reviewer

Use this skill when evaluating whether dbt models should be decomposed into stable dimensions, facts, bridge tables, survivor candidates, or conformed lookup tables.

## Required Inputs

Read `docs/05_dbt_model_layering_and_normal_form_evolution_plan.md`, `metadata/model_specs/dbt/dbt_model_layering_contract.yaml`, `metadata/model_specs/silver/**`, `metadata/model_specs/mappings/provider_to_silver_matrix.yaml`, `metadata/model_specs/evolution/V0_5/**`, `.agent/spec_templates/normalization_probe.template.yaml`, `dbt/models/derived/**`, `dbt/models/data_quality/**`, `reports/dataq_drifts/**`, and the latest dbt artifacts under `dbt/target/` only when generated by approved local validation.

## Review Heuristics

Evaluate one entity family at a time. Start from provider-filtered source samples and local PostgreSQL profiling before proposing decomposition. Do not normalize by intuition alone.

Check the normal forms in order:

```text
1NF: no nested values, repeated groups, encoded lists, mixed scalar/array fields, or non-atomic values.
2NF: every non-key field depends on the full grain key, not only provider, member, code, date, or another partial key.
3NF: descriptive attributes depend on the entity key, not on another descriptive attribute that should be a dimension.
4NF: independent multivalued facts are separated, such as member coverage periods, medication code variants, observation components, or provider metadata.
```

Name the grain explicitly before accepting any table. If a table mixes member grain, event grain, period grain, code grain, and provider grain, propose a normalization probe before changing durable models.

## Decomposition Rules

Create lookup or dimension tables when values are reused, governed by a domain, need metadata, or should not be repeated in every fact. Examples include providers, gender domains, record statuses, coverage statuses, code systems, units, amount source types, and timezone contracts.

Create bridge tables when a relationship is many-to-many or when one source record creates multiple independent normalized rows. Create survivor candidates when duplicate business keys exist but history must be preserved.

Do not drop data to satisfy normal form. Preserve raw lineage, source row ids, provider scope, review batch id, raw value fields when needed, normalized value fields, quality status, unresolved flags, and enough counts to reconcile back to Silver.

## Probe And HITL Rules

Use a normalization probe when nested data, duplicate keys, cross-entity fields, partial dependencies, transitive dependencies, multivalued dependencies, unsafe casts, or sparse ambiguous fields are found. The probe must define candidate grain, keys, row accounting, before/after counts, rollback target, tests, and unresolved HITL questions.

Stop for HITL only when the probe cannot determine ownership, meaning, clinical interpretation, financial interpretation, business impact, or privacy risk. Do not ask HITL to approve deterministic cleanup already governed by contract.

## QA Rules

Each accepted normalization change needs tests for row accounting, key uniqueness at the declared grain, non-null required keys, referential integrity, provider isolation, duplicate survivorship when applicable, and no loss of lineage. Use full local `data_500k` PostgreSQL data when the active plan requires exhaustive validation.

Normal form review is not done until dbt lineage shows the new dependency path, schema YAML documents the grain, and the plan or trace log records why the model belongs in `silver/normalized`, `silver/conformed`, `silver/dq`, `gold/facts`, `gold/dimensions`, or `gold/marts`.
